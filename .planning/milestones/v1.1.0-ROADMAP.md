# Milestone v1.1.0: Visual Provider Reliability and Downstream Hardening

**Status:** ✅ SHIPPED 2026-03-28
**Phases:** 14-15
**Total Plans:** 4

## Overview

`v1.1.0` 的目标不是扩主链，而是把已经打通的 `keep-list -> visual-review -> export` 下游链路收稳。这个 milestone 聚焦两件事：先把视觉 provider 的配置来源、预检和错误语义做清楚，再用真实 `MINISO` keep-list 跑出一轮 non-error 的 bounded visual review，证明当前 `openai` 路径已经可以真实落到导出。

## Phases

### Phase 14: Stabilize visual provider config and preflight

**Goal**: 修复当前视觉 provider 的鉴权解析路径，让 bounded run 在真正进视觉前就能暴露配置问题，并把 provider 诊断信息写进 summary。
**Depends on**: Phase 13
**Plans**: 2 plans

Plans:

- [x] 14-01: Add backend vision provider snapshot, preflight, and early failure diagnostics
- [x] 14-02: Surface vision diagnostics in runner summaries/docs and leave a bounded diagnostic validation artifact

**Details:**
当前 milestone 为 backend 建立了视觉 provider 诊断契约，包括 provider snapshot、preflight 和 structured early failure。`/api/health`、`POST /api/jobs/visual-review`、keep-list runner 和 smoke runner 现在都会暴露 `vision_preflight` 与 provider 相关字段，operator 能明确看到 backend 认出了哪些 provider、key/base URL/model 从哪里来、为什么 runnable 或为什么 early fail。README 也同步更新，明确了视觉配置真正吃的是 backend 进程环境与仓库根目录 `.env.local`，而不是 runner 的 `--env-file`。

### Phase 15: Validate non-error bounded visual review flow

**Goal**: 基于修复后的 provider 配置，对真实 keep-list 执行一轮 non-error 的 bounded visual review，并把 operator path 文档化。
**Depends on**: Phase 14
**Plans**: 2 plans

Plans:

- [x] 15-01: Add deterministic provider selection and lightweight live auth checks for bounded visual runs
- [x] 15-02: Execute the real non-error bounded visual validation and document the final operator path

**Details:**
在 Phase 15 中，backend 和 keep-list runner 增加了显式 `provider` 选择与 live probe 能力。真实 `MINISO` bounded run 通过 `--vision-provider openai` 覆盖了 `staging -> scrape -> prescreen -> visual-review -> export` 全链，summary 明确记录了 `vision_probe.success = true`、`scrape 1/1`、`prescreen_pass_count = 1`、visual `pass = 1 / error = 0`，并在最终导出里给出 `_sophiesilva_` 的真实视觉结果 `Pass`。这说明此前的 `auth_not_found` 不能再被概括成“现在 apikey 没配好”，更准确的结论是旧 run 的 provider 路径或请求面不稳定，而当前显式 `openai` 路径已经 real run 成功。

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Decision: 由 backend 统一拥有视觉 provider snapshot / preflight 契约，再由 runner 直接透传到 summary 与文档
- Decision: bounded visual run 支持显式 `--vision-provider`，避免 provider 选择再受隐式顺序和旧状态影响
- Decision: 在真实 visual run 前增加 lightweight live probe，把“配置读到了什么”和“真实请求是否可通”拆开验证

**Issues Resolved:**

- 视觉 provider 缺失或错误配置不再只在尾部表现为模糊失败，能在 preflight 或 visual start 早期返回结构化错误
- keep-list / smoke summary 现在会保留 provider 诊断字段，operator 可以直接基于 artifact 定位问题
- 已对真实 `MINISO` keep-list 完成一轮 non-error bounded visual validation，当前 `openai` 路径可从 scrape 跑到 export

**Issues Deferred:**

- 未在归档前补做专门的 milestone audit 文件
- `quan2go` / `lemonapi` 还没有 live proof run
- 当前 proof run 仍然是 bounded 的 `instagram 1`，不是全量批次稳定性证明

**Technical Debt Incurred:**

- 旧的 workbook / dashboard / project-home 流程仍依赖外部全量 `email` 项目
- 报价结果还没有正式接入 `筛号` 运行态或最终导出链
- 对 OpenAI-compatible provider 的长期稳定性证明仍不完整，目前只保留了 `openai` 的真实成功样本
