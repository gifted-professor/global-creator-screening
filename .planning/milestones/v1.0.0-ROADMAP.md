# Roadmap: chuhaihai

## Overview

这个仓库会分阶段吸收原来分散在 sibling 目录里的创作者筛选工作流模块。前半段先补齐规划骨架，并完成飞书桥接、邮件抓取、模板解析和报价抽取迁移；当前已经把 `筛号` 后端主链路也迁进当前仓库。

## Phases

- [x] **Phase 1: Bootstrap Planning** - 建立最小 GSD 项目骨架与仓库边界
- [x] **Phase 2: Migrate Feishu Bridge** - 迁入飞书内容获取代码、测试与 env 对齐
- [x] **Phase 3: Prepare Next Consolidation** - 验证迁移结果并为后续 sibling 模块迁移做准备
- [x] **Phase 4: Integrate email_sync into current repo** - 迁入任务驱动邮件抓取核心链路并保持桥接兼容
- [x] **Phase 5: Integrate workbook template parser into current repo** - 迁入模板解析能力，并打通下载后的立即解析
- [x] **Phase 6: Integrate task-driven mail sync bridge** - 把任务上传映射结果直接接到当前仓库本地邮件抓取入口
- [x] **Phase 7: Integrate creator enrichment into current repo** - 把达人库匹配与报价抽取业务层迁入当前仓库并做真实验证
- [x] **Phase 8: Integrate screening backend into current repo** - 把 `筛号` 后端、配置、脚本和 env 对齐迁入当前仓库
- [x] **Phase 9: Bridge upstream outputs into screening backend state** - 把模板 rulespec 和达人名单写入当前筛号后端输入状态

- [x] **Phase 11: Resolve duplicate creator matches with group-level LLM review** - 对重复 `last_mail` 组按邮件组而不是达人行做一次 LLM 归属裁决
- [x] **Phase 12: Align production duplicate-review pipeline with keep-list outputs** - 把 sample-first duplicate review sidecar 流程对齐到正式 keep-list 产物链
- [x] **Phase 13: Wire keep-list outputs into screening pipeline** - 把 `*_llm_reviewed_keep.xlsx` 正式接回当前 `筛号` 的输入准备、抓取、预筛、视觉复核与导出链
